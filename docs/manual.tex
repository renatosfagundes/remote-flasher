% === 1. Introdução ===============================================
\section{Introdução}

O \textbf{Remote Firmware Flasher} é uma aplicação desktop (PySide6) que permite programar placas Arduino remotamente, através de conexão VPN e SSH. Foi desenvolvido para a Residência Tecnológica do CIn/UFPE, permitindo que alunos gravem firmware nas placas do laboratório sem precisar estar fisicamente presente.

\subsection{Funcionalidades}

\begin{itemize}[nosep]
  \item Conexão VPN (SSTP) diretamente pela aplicação
  \item Upload e gravação de firmware (\texttt{.hex}) via SCP + \texttt{avrdude}
  \item Reset das placas via scripts PowerShell remotos
  \item Até 4 terminais serial simultâneos com envio de comandos
  \item Terminal SSH para executar comandos no PC remoto
  \item Upload de arquivos e pastas via SFTP com barra de progresso
  \item Câmera ao vivo para visualizar as placas
  \item Configuração automática do ambiente de desenvolvimento (Setup)
  \item Guias de integração com Notepad++ e VSCode
\end{itemize}

% === 2. Instalação ===============================================
\section{Instalação}

\subsection{Executável (recomendado)}

Basta baixar o arquivo \texttt{RemoteFlasher.exe} e executar. Não é necessário instalar Python ou qualquer dependência.

\subsection{Código-fonte}

Para executar a partir do código-fonte:

\begin{codigobox}
git clone https://github.com/renatosfagundes/remote-flasher.git\\
cd remote-flasher\\
pip install -r requirements.txt\\
cp secrets.example.py secrets.py   \# editar com as credenciais do lab\\
python src/main.py
\end{codigobox}

\textbf{Dependências:} Python 3.9+, PySide6, paramiko, requests.

\subsection{Configurando as credenciais (\texttt{secrets.py})}

O arquivo \texttt{secrets.py} contém os endereços IP, usuários e senhas SSH dos PCs do laboratório. Esse arquivo \textbf{não é incluído no repositório} por segurança. Para obtê-lo, entre em contato com \textbf{Davi} ou \textbf{Renato}.

% === 3. Configuração do Ambiente =================================
\section{Configuração do Ambiente de Desenvolvimento}

A aba \textbf{Setup} do aplicativo automatiza toda a configuração do ambiente necessário para compilar e gravar firmware usando o Trampoline RTOS.

\subsection{O que é instalado automaticamente}

\begin{center}
\begin{tabularx}{\textwidth}{lX}
  \toprule
  \textbf{Componente} & \textbf{Descrição} \\
  \midrule
  Arduino CLI       & Ferramenta de linha de comando para compilar e gravar firmware Arduino \\
  AVR Toolchain     & Compilador GCC para microcontroladores AVR (avr-gcc, avr-objcopy, etc.) \\
  Trampoline RTOS   & Sistema operacional de tempo real compatível com OSEK/AUTOSAR \\
  goil              & Compilador OIL (OSEK Implementation Language) do Trampoline \\
  Arduino Core      & Bibliotecas do Arduino para uso com Trampoline \\
  Template Nano     & Template de configuração do Trampoline para Arduino Nano \\
  MCP\_CAN          & Biblioteca para comunicação CAN via módulo MCP2515 \\
  \bottomrule
\end{tabularx}
\end{center}

\subsection{Executando o Setup}

\begin{enumerate}[nosep]
  \item Abra a aba \textbf{Setup}
  \item Os indicadores mostram o status de cada componente (verde = instalado, vermelho = ausente)
  \item Clique em \textbf{``Run Full Setup''} para instalar tudo automaticamente
  \item Acompanhe o progresso no log à esquerda
  \item Após a conclusão, abra um \textbf{novo terminal} para que as variáveis PATH sejam reconhecidas
\end{enumerate}

\begin{notabox}[Importante]
O setup requer conexão com a internet para baixar os componentes. O processo pode levar alguns minutos na primeira execução. Execuções subsequentes são rápidas pois detectam o que já está instalado.
\end{notabox}

\subsection{Verificação manual}

Após o setup, abra um novo terminal e verifique:

\begin{codigobox}
arduino-cli version\\
avr-gcc --version\\
goil --version
\end{codigobox}

\subsection{Estrutura de diretórios}

Todos os componentes são instalados em \texttt{C:\textbackslash ESA}:

\begin{codigobox}
C:\textbackslash ESA\textbackslash\\
\quad arduino\_cli\textbackslash bin\textbackslash\quad\quad\# Arduino CLI\\
\quad avr\_tools\textbackslash\quad\quad\quad\quad\quad\quad\# AVR GCC Toolchain + goil\\
\quad trampoline\textbackslash\quad\quad\quad\quad\quad\# Trampoline RTOS\\
\quad\quad examples\textbackslash avr\textbackslash\quad\quad\# Exemplos (arduinoUno, arduinoNano)\\
\quad\quad goil\textbackslash templates\textbackslash\quad\# Templates de configuração\\
\quad\quad opt\textbackslash devel\textbackslash\quad\quad\quad\# Diretório de desenvolvimento\\
\end{codigobox}

% === 7. Configuração CAN ==========================================
\section{Configuração de Rede CAN}

A aba \textbf{CAN} permite configurar em qual barramento CAN (CAN1 ou CAN2) cada placa está conectada. A placa ANEB possui duas redes CAN independentes, e cada placa pode ser comutada entre elas via comando AT no microcontrolador de controle.

\subsection{Topologia visual}

A parte superior da aba mostra um diagrama em tempo real com dois barramentos (CAN~1 em verde, CAN~2 em laranja) e as placas conectadas a cada um. Placas sem seletor CAN aparecem em cinza com ``?'' entre os dois barramentos.

\begin{figure}[H]
  \centering
  \includegraphics[width=\textwidth]{can_topology.png}
  \caption{Topologia CAN --- Placas 01 e 02 em CAN~1, Placas 03 e 04 em CAN~2.}
\end{figure}

\subsection{Selecionando a rede}

\begin{enumerate}[nosep]
  \item Selecione o PC no menu \textbf{Computer}
  \item Para cada placa, selecione \textbf{CAN~1} ou \textbf{CAN~2}
  \item Clique em \textbf{Apply} para enviar o comando de comutação
  \item O log confirma com \texttt{MCU1 OK}
\end{enumerate}

\subsection{Presets rápidos}

Botões de configuração rápida permitem colocar todas as placas no mesmo barramento ou dividir em grupos com um clique. O botão \textbf{Detect Boards} consulta a identidade e firmware de cada placa.

\begin{figure}[H]
  \centering
  \includegraphics[width=\textwidth]{can_applied.png}
  \caption{Todas as placas configuradas --- status verde confirmando a aplicação.}
\end{figure}

\begin{notabox}
Para que duas ECUs se comuniquem via CAN, ambas devem estar no \textbf{mesmo barramento}. Se a comunicação não funciona, verifique se ambas estão em CAN~1 (ou ambas em CAN~2).
\end{notabox}

% === 8. Terminal Serial ==========================================
\section{Terminal Serial}

A aba \textbf{Serial} permite monitorar até 4 conexões seriais simultaneamente e enviar comandos.

\subsection{Layout adaptativo}

\begin{itemize}[nosep]
  \item \textbf{1 conexão:} painel único, área completa
  \item \textbf{2 conexões:} lado a lado (divisor horizontal)
  \item \textbf{3 conexões:} 2 em cima + 1 embaixo
  \item \textbf{4 conexões:} grade 2$\times$2
\end{itemize}

Clique em \textbf{``+ Add Serial''} para adicionar uma nova conexão (máximo 4). Cada painel pode ser fechado pelo botão \textbf{X} (que desconecta automaticamente).

\begin{figure}[H]
  \centering
  \includegraphics[width=\textwidth]{serial_1panel.png}
  \caption{Um painel serial --- saída de tarefas RTOS com Virtual I/O na parte inferior.}
\end{figure}

\begin{figure}[H]
  \centering
  \includegraphics[width=\textwidth]{serial_2panels.png}
  \caption{Dois painéis lado a lado --- TX e RX monitorados simultaneamente.}
\end{figure}

\subsection{Enviando comandos}

Cada painel possui um campo de texto na parte inferior para enviar comandos pela porta serial. Digite o comando e pressione \textbf{Enter} ou clique em \textbf{Send}. O campo só fica ativo quando a conexão está aberta.

\subsection{Autoscroll}

Desmarque a caixa \textbf{``Autoscroll''} para congelar a posição do scroll enquanto os dados continuam chegando --- útil para analisar mensagens CAN sem perder a posição.

\begin{figure}[H]
  \centering
  \includegraphics[width=\textwidth]{serial_autoscroll.png}
  \caption{Autoscroll desativado no painel esquerdo enquanto dados CAN com sequenciador continuam chegando.}
\end{figure}

\subsection{Exclusão de portas}

Quando um painel está conectado a uma porta COM, essa porta não aparece nos menus dos outros painéis, evitando conflitos.

\subsection{Upload do serialterm.py}

Na primeira utilização, clique em \textbf{``Upload serialterm.py''} em qualquer painel serial \textbf{antes} de conectar. Isso envia a versão bidirecional do script para o PC remoto. Só é necessário fazer isso uma vez --- o arquivo é reutilizado por todas as conexões do mesmo PC.

\begin{notabox}[Importante]
Se o script remoto for antigo (versão somente leitura), os controles Virtual I/O não funcionarão. Sempre faça o upload ao usar uma versão nova do Remote Flasher.
\end{notabox}

\subsection{Virtual I/O --- Controles integrados}

Cada painel serial possui um painel de \textbf{Virtual I/O} na parte inferior com:

\begin{itemize}[nosep]
  \item \textbf{4 botões} (B1--B4) --- redondo, 3D, com cores distintas. Clique para pressionar/soltar.
  \item \textbf{4 LEDs} (L1--L4) --- indicadores que o firmware controla remotamente. O brilho varia de 0 (apagado) a 255 (máximo).
  \item \textbf{2 potenciômetros} (POT1, POT2) --- sliders de 0 a 1023 que enviam valores ao firmware.
\end{itemize}

Os controles ficam ativos somente quando a conexão serial está aberta. Ao desconectar, todos voltam ao estado inicial.

\begin{figure}[H]
  \centering
  \includegraphics[width=\textwidth]{serial_virtualio.png}
  \caption{Virtual I/O em ação --- potenciômetros controlando velocidade CAN, LEDs indicando TX/RX.}
\end{figure}

Para utilizar o Virtual I/O no firmware, inclua a biblioteca \texttt{VirtualIO.h} (disponível em \texttt{arduino\_lib/VirtualIO/}). Veja a Seção~\ref{sec:virtualio} para detalhes do protocolo e exemplos.

\subsection{Plugins necessários}

Instale via \textbf{Plugins $\rightarrow$ Plugins Admin}:
\begin{itemize}[nosep]
  \item \textbf{NppExec} --- execução de comandos e scripts
  \item \textbf{SourceCookifier} --- navegação de código
\end{itemize}

Instale manualmente o \textbf{NppConsole} (baixe de \url{https://sourceforge.net/projects/nppconsole/}).

\subsection{Exemplo: Compilar e gravar um projeto Trampoline}

Crie o seguinte script em \texttt{Plugins $\rightarrow$ NppExec $\rightarrow$ Execute}:

\begin{lstlisting}[caption={Script NppExec: trampoline\_build\_and\_flash.scp}]
// [Trampoline] Build and Flash
NPP_SAVEALL
cd $(CURRENT_DIRECTORY)

// Run goil to generate build files
goil --target=avr/arduino/nano --templates=../../../../goil/templates/ $(NAME_PART).oil

// Compile
cmd /c python make.py

// Flash (change COM port as needed)
INPUTBOX "Serial port:":"COM": "COM25"
set local commport = $(INPUT)
arduino-cli upload -b arduino:avr:nano:cpu=atmega328old -p $(commport) -v -i image.hex -t
\end{lstlisting}

\begin{dicabox}
Para criar um atalho de teclado: \textbf{Plugins $\rightarrow$ NppExec $\rightarrow$ Advanced Options}, associe o script a um item de menu, depois em \textbf{Settings $\rightarrow$ Shortcut Mapper} atribua uma tecla.
\end{dicabox}

% === 13. Referência OIL ===========================================
\section{Referência Rápida --- Arquivo OIL}

O arquivo \texttt{.oil} descreve a configuração do RTOS em linguagem OIL (OSEK Implementation Language). Abaixo estão os padrões mais usados.

\subsection{Estrutura básica}

\begin{lstlisting}[language=OIL]
OIL_VERSION = "2.5" : "nome_projeto";

CPU test {
  OS config {
    STATUS = STANDARD;
    BUILD = TRUE {
      TRAMPOLINE_BASE_PATH = "../../../";
      APP_NAME = "image";
      APP_SRC = "meu_arquivo.cpp";
      // APP_SRC = "VirtualIO.cpp";  // se usar Virtual I/O
      CPPCOMPILER = "avr-g++";
      COMPILER = "avr-gcc"; LINKER = "avr-gcc";
      ASSEMBLER = "avr-gcc"; COPIER = "avr-objcopy";
      SYSTEM = PYTHON;
      LIBRARY = serial;
      // LIBRARY = spi;      // se usar CAN
      // LIBRARY = mcp_can;  // se usar CAN
    };
    SYSTEM_CALL = TRUE;
  };
  APPMODE stdAppmode {};

  // ... TASKs, ALARMs, RESOURCEs aqui ...
};
\end{lstlisting}

\subsection{TASK --- Declaração de tarefa}

\begin{lstlisting}[language=OIL]
  TASK NomeDaTask {
    PRIORITY = 10;        // maior = mais prioritário
    AUTOSTART = FALSE;    // TRUE se deve iniciar automaticamente
    ACTIVATION = 1;       // máximo de ativações pendentes
    SCHEDULE = FULL;      // FULL = preemptivo, NON = cooperativo
    STACKSIZE = 256;      // tamanho da pilha em bytes
    // RESOURCE = nomeRecurso;  // se usar recurso compartilhado
  };
\end{lstlisting}

No código C++, a task é declarada com \texttt{TASK(NomeDaTask) \{ ... \}}. \textbf{Sempre termine com \texttt{TerminateTask();}}.

\subsection{ALARM --- Ativação periódica}

\begin{lstlisting}[language=OIL]
  ALARM nomeAlarme {
    COUNTER = SystemCounter;
    ACTION = ACTIVATETASK { TASK = NomeDaTask; };
    AUTOSTART = TRUE {
      ALARMTIME = 500;    // primeiro disparo (ms)
      CYCLETIME = 500;    // período de repetição (ms)
      APPMODE = stdAppmode;
    };
  };
\end{lstlisting}

\textbf{API para mudar período em tempo de execução:}
\begin{lstlisting}[language=C++]
CancelAlarm(nomeAlarme);                    // cancela alarme ativo
SetRelAlarm(nomeAlarme, delay, cycle);      // reprograma (ms)
\end{lstlisting}

\textit{Ver exemplo \texttt{03\_alarm\_dynamic} na pasta \texttt{examples/}.}

\subsection{RESOURCE --- Exclusão mútua}

\begin{lstlisting}[language=OIL]
  RESOURCE nomeRecurso {
    RESOURCEPROPERTY = STANDARD;  // ou INTERNAL
  };
\end{lstlisting}

\textbf{Uso no código C++:}
\begin{lstlisting}[language=C++]
TASK(MinhaTask) {
    GetResource(nomeRecurso);
    // ... seção crítica (acesso exclusivo) ...
    ReleaseResource(nomeRecurso);
    TerminateTask();
}
\end{lstlisting}

Declare \texttt{RESOURCE = nomeRecurso;} na TASK que o utiliza (no .oil). Recursos impedem inversão de prioridade usando o protocolo \emph{Priority Ceiling}.

\textit{Ver exemplo \texttt{04\_resource\_prodcons} na pasta \texttt{examples/}.}

% === 14. Referência CAN ===========================================
\section{Referência Rápida --- Comunicação CAN}

\subsection{Configuração no arquivo OIL}

Para usar CAN, adicione as bibliotecas \texttt{spi} e \texttt{mcp\_can}:

\begin{lstlisting}[language=OIL]
      LIBRARY = serial;
      LIBRARY = spi;
      LIBRARY = mcp_can;
\end{lstlisting}

\subsection{Inicialização}

\begin{lstlisting}[language=C++]
#include "Board.h"
#include <mcp_can.h>

MCP_CAN CAN1(CAN1_CS);  // chip select da placa

void setup() {
    Serial.begin(115200);
    pinMode(CAN_INT, INPUT);
    while (CAN1.begin(MCP_ANY, CAN_500KBPS, MCP_8MHZ) != CAN_OK) {
        delay(200);
    }
    CAN1.setMode(MCP_NORMAL);
}
\end{lstlisting}

\subsection{Enviar mensagem}

\begin{lstlisting}[language=C++]
byte data[8] = {0x01, 0x02, 0x03, 0x04, 0x05, 0x06, 0x07, 0x08};

// Frame standard (11 bits)
CAN1.sendMsgBuf(0x100, CAN_STDID, 8, data);

// Frame extended (29 bits) — usado em J1939
CAN1.sendMsgBuf(0x18FEF100, CAN_EXTID, 8, data);
\end{lstlisting}

\subsection{Receber mensagem}

\begin{lstlisting}[language=C++]
TASK(Receiver) {
    if (!digitalRead(CAN_INT)) {    // há mensagem no buffer?
        unsigned char dlc = 0;
        unsigned char data[8];
        long unsigned int id;

        CAN1.readMsgBuf(&id, &dlc, data);

        unsigned long canId = id & 0x1FFFFFFFUL;  // 29-bit ID
        // data[0..dlc-1] contém os bytes da mensagem
    }
    TerminateTask();
}
\end{lstlisting}

\subsection{Encapsular/Desencapsular sinais}

Para encapsular um valor em bytes do frame CAN:

\begin{lstlisting}[language=C++]
// Encapsular: valor float -> bytes CAN com resolução
float velocidade_kmh = 120.5;
float resolucao = 1.0 / 256.0;  // 1/256 km/h por bit
unsigned int raw = (unsigned int)(velocidade_kmh / resolucao);
data[1] = (raw >> 8) & 0xFF;  // byte alto
data[2] = raw & 0xFF;          // byte baixo
\end{lstlisting}

Para desencapsular:

\begin{lstlisting}[language=C++]
// Desencapsular: bytes CAN -> valor float
unsigned int raw = ((unsigned int)data[1] << 8) | data[2];
float velocidade_kmh = raw * resolucao;
\end{lstlisting}

% === 15. Referência Board.h =======================================
\section{Referência Rápida --- Board.h (Pinos da Placa)}

O arquivo \texttt{Board.h} define os pinos do \emph{Automotive Network Evaluation Board} (ANEB v1.1). Cada ECU possui:

\begin{center}
\small
\begin{tabularx}{\textwidth}{llX}
  \toprule
  \textbf{Tipo} & \textbf{Pinos (ECU1)} & \textbf{Função} \\
  \midrule
  Entrada digital & \texttt{ECU1\_DIN1..DIN4} & Pushbuttons (\texttt{digitalRead}) \\
  Saída digital   & \texttt{ECU1\_DOUT1, DOUT2} & LEDs (\texttt{digitalWrite}) \\
  Buzzer          & \texttt{ECU1\_BUZZER} & Buzzer (só ECU1) \\

  CAN interrupt   & \texttt{ECU1\_CAN1\_INT} & MCP2515 INT (\texttt{CAN\_INT}) \\
  \bottomrule
\end{tabularx}
\end{center}

Para usar outra ECU, substitua o prefixo: \texttt{ECU2\_DIN1}, \texttt{ECU3\_AIN2}, etc. A pinagem física é a mesma para todas as ECUs.

% === 16. Variáveis Compartilhadas ================================
\section{Referência Rápida --- Variáveis Compartilhadas entre Tasks}

Tasks no Trampoline podem compartilhar dados através de variáveis globais. Use \texttt{volatile} para garantir que o compilador não otimize o acesso:

\begin{lstlisting}[language=C++]
static volatile int sharedValue = 0;  // compartilhada entre tasks

TASK(Producer) {
    sharedValue = analogRead(ECU1_AIN1);  // escreve
    TerminateTask();
}

TASK(Consumer) {
    int local = sharedValue;  // lê
    Serial.println(local);
    TerminateTask();
}
\end{lstlisting}

\begin{notabox}[Cuidado com inconsistência]
Se uma task de alta prioridade interrompe outra durante a escrita de uma variável multi-byte, o leitor pode ver um valor parcialmente atualizado. Use \texttt{RESOURCE} para proteger seções críticas:
\end{notabox}

\begin{lstlisting}[language=C++]
// No .oil: RESOURCE dataRes { RESOURCEPROPERTY = STANDARD; };

TASK(Writer) {
    GetResource(dataRes);
    sharedSpeed = newSpeed;    // escrita protegida
    sharedRPM = newRPM;
    ReleaseResource(dataRes);
    TerminateTask();
}

TASK(Reader) {
    int s, r;
    GetResource(dataRes);
    s = sharedSpeed;           // leitura protegida
    r = sharedRPM;
    ReleaseResource(dataRes);
    Serial.print("V="); Serial.print(s);
    Serial.print(" R="); Serial.println(r);
    TerminateTask();
}
\end{lstlisting}

\textit{Ver exemplo \texttt{04\_resource\_prodcons} na pasta \texttt{examples/}.}

Este exemplo demonstra duas ECUs trocando mensagens CAN usando tarefas do Trampoline e controles \textbf{Virtual I/O}. O transmissor usa POT1/POT2 do painel GUI para definir valores e os envia via CAN. O receptor decodifica e mostra o resultado nos LEDs virtuais.

#include "tpl_os.h"
#include "Arduino.h"
#include "Board.h"
#include "VirtualIO.h"
#include <mcp_can.h>

#define POT_ID 0x200
MCP_CAN CAN1(CAN1_CS);

void setup() {
  Serial.begin(115200);
  vioInit();
  while (CAN1.begin(MCP_ANY, CAN_500KBPS, MCP_8MHZ) != CAN_OK) {
    delay(200);
  }
  CAN1.setMode(MCP_NORMAL);
  Serial.println("CAN TX ready. Use POT1 and POT2.");
}

TASK(SendPotValue) {
  vioUpdate();
  int pot1 = vAnalogRead(VPOT1);
  int pot2 = vAnalogRead(VPOT2);
  byte data[4] = { highByte(pot1), lowByte(pot1),
                    highByte(pot2), lowByte(pot2) };
  byte ret = CAN1.sendMsgBuf(POT_ID, CAN_STDID, 4, data);
  if (ret == CAN_OK) {
    Serial.print("TX pot1=");
    Serial.print(pot1);
    Serial.print(" pot2=");
    Serial.println(pot2);
  }
  TerminateTask();
}
\end{lstlisting}

\subsection{ECU Receptora (can\_pot\_rx)}

\begin{lstlisting}[language=C++,caption={can\_pot\_rx.cpp (trecho relevante)}]
#include "VirtualIO.h"

TASK(ReceiveAndAct) {
  vioUpdate();
  if (!digitalRead(CAN_INT)) {
    unsigned char dlc = 0;
    unsigned char data[8];
    long unsigned int id;
    CAN1.readMsgBuf(&id, &dlc, data);

    if ((id & 0x1FFFFFFF) == POT_ID && dlc >= 4) {
      int pot1 = (data[0] << 8) | data[1];
      int pot2 = (data[2] << 8) | data[3];

      // LEDs virtuais mostram os valores recebidos
      vLedWrite(VLED1, pot1 / 4);  // vermelho = pot1
      vLedWrite(VLED2, pot2 / 4);  // azul = pot2
      vLedWrite(VLED3, 255);       // verde = CAN ativo

      Serial.print("RX pot1=");
      Serial.print(pot1);
      Serial.print(" pot2=");
      Serial.println(pot2);
    }
  }
  TerminateTask();
}
\end{lstlisting}

% === 14. Virtual I/O =============================================
\section{Virtual I/O --- Controles Remotos}
\label{sec:virtualio}

Cada painel serial possui um painel de \textbf{Virtual I/O} com 4 botões e 2 potenciômetros virtuais. Esses controles enviam comandos pela porta serial, permitindo simular entradas físicas sem acesso à placa.

\subsection{Protocolo}

\begin{center}
\begin{tabular}{lll}
  \toprule
  \textbf{Comando} & \textbf{Significado} & \textbf{Exemplo} \\
  \midrule
  \texttt{!B1:1} & Botão 1 pressionado & Envia ao pressionar BTN1 \\
  \texttt{!B1:0} & Botão 1 solto & Envia ao soltar BTN1 \\
  \texttt{!P1:512} & Potenciômetro 1 = 512 & Envia ao mover slider POT1 \\
  \texttt{!P2:0} & Potenciômetro 2 = 0 & Faixa: 0--1023 \\
  \bottomrule
\end{tabular}
\end{center}

O prefixo \texttt{!} diferencia comandos VIO de texto normal. Os sliders possuem \emph{throttle} de 50\,ms para evitar sobrecarga na serial.

\subsection{LEDs Virtuais (saídas)}

Além dos controles de entrada, cada painel possui 4 \textbf{LEDs virtuais} (L1--L4) que o Arduino pode controlar. Os LEDs aparecem entre os botões e os potenciômetros, com cores correspondentes (vermelho, azul, amarelo, verde).

O firmware envia comandos pela serial para atualizar os LEDs:

\begin{center}
\begin{tabular}{lll}
  \toprule
  \textbf{Comando} & \textbf{Significado} & \textbf{Exemplo} \\
  \midrule
  \texttt{!L1:1} & LED 1 ligado & \texttt{vLedWrite(VLED1, 1)} \\
  \texttt{!L1:0} & LED 1 desligado & \texttt{vLedWrite(VLED1, 0)} \\
  \texttt{!L2:128} & LED 2 meio brilho & \texttt{vLedWrite(VLED2, 128)} \\
  \bottomrule
\end{tabular}
\end{center}

A função \texttt{vLedWrite()} só envia o comando quando o valor muda, evitando sobrecarregar a serial.
Os comandos \texttt{!L} são interceptados pelo Remote Flasher e \textbf{não aparecem no log} --- apenas atualizam os indicadores visuais.

\subsection{Biblioteca Arduino: VirtualIO}

Para usar os controles virtuais, copie os arquivos \texttt{VirtualIO.h} e \texttt{VirtualIO.cpp} (disponíveis em \texttt{arduino\_lib/VirtualIO/}) para a pasta do seu projeto. A biblioteca ocupa apenas 6 bytes de RAM e tem overhead desprezível.

\begin{lstlisting}[language=C++,caption={API da biblioteca VirtualIO}]
#include "VirtualIO.h"

void setup() {
  Serial.begin(115200);
  vioInit();  // inicializa o sistema de I/O virtual
}

TASK(MinhaTask) {
  vioUpdate();  // lê comandos pendentes da serial (leve, retorna imediato)

  // Ler botões virtuais (retorna 0 ou 1)
  if (vRead(VBTN1)) { /* BTN1 está pressionado */ }
  if (vRead(VBTN3)) { /* BTN3 está pressionado */ }

  // Ler potenciômetros virtuais (retorna 0–1023)
  int valor = vAnalogRead(VPOT1);
}
\end{lstlisting}

\begin{notabox}[Importante]
Chame \texttt{vioUpdate()} no início de qualquer task que use os controles virtuais. A função é leve --- retorna imediatamente se não há dados na serial.
\end{notabox}

\begin{lstlisting}[language=C++,caption={vio\_led.cpp}]
#include "tpl_os.h"
#include "Arduino.h"
#include "VirtualIO.h"

#define LED_PIN 3  // PWM pin (physical)

void setup() {
  Serial.begin(115200);
  vioInit();
  pinMode(LED_PIN, OUTPUT);
  Serial.println("VIO LED example ready.");
  Serial.println("BTN1=toggle, POT1=brightness, BTN2=LED2 blink");
}

TASK(ControlLed) {
  vioUpdate();

  if (vRead(VBTN1)) {
    // BTN1 pressed: physical + virtual LED on, brightness from POT1
    int brightness = vAnalogRead(VPOT1) / 4;  // 0-1023 -> 0-255
    analogWrite(LED_PIN, brightness);
    vLedWrite(VLED1, brightness);    // mirror on GUI LED 1
  } else {
    analogWrite(LED_PIN, 0);
    vLedWrite(VLED1, 0);             // GUI LED 1 off
  }

  // BTN2: toggle GUI LED 2 (no physical counterpart)
  vLedWrite(VLED2, vRead(VBTN2) ? 255 : 0);

  // LED 3 shows POT2 as a bar (brightness proportional to pot)
  vLedWrite(VLED3, vAnalogRead(VPOT2) / 4);
}
\end{lstlisting}

\textbf{Para testar:}
\begin{enumerate}[nosep]
  \item Compile e grave o firmware no Arduino
  \item Abra a conexão serial no Remote Flasher
  \item Pressione \textbf{BTN1} --- o LED físico acende e o \textbf{LED 1} virtual fica vermelho
  \item Mova \textbf{POT1} --- o brilho do LED físico e virtual muda em tempo real
  \item Pressione \textbf{BTN2} --- o \textbf{LED 2} virtual (azul) acende
  \item Mova \textbf{POT2} --- o \textbf{LED 3} virtual (amarelo) muda de brilho
\end{enumerate}

\subsection{Exemplo 2: CAN com controles virtuais}

Este exemplo envia mensagens CAN cujo conteúdo é controlado pelos botões e potenciômetros virtuais. A ECU transmissora usa BTN1--BTN4 como flags de status e POT1/POT2 como valores analógicos no frame CAN.

\begin{lstlisting}[language=C++,caption={vio\_can\_tx.cpp (trecho relevante)}]
#include "tpl_os.h"
#include "Arduino.h"
#include "Board.h"
#include "VirtualIO.h"

#define MSG_ID 0x300
MCP_CAN CAN1(CAN1_CS);

void setup() {
  Serial.begin(115200);
  vioInit();
  while (CAN1.begin(MCP_ANY, CAN_500KBPS, MCP_8MHZ) != CAN_OK) {
    delay(200);
  }
  CAN1.setMode(MCP_NORMAL);
  Serial.println("VIO CAN TX ready. Use BTN1-4 and POT1-2.");
}

TASK(SendFrame) {
  vioUpdate();

  byte data[8];
  // Bytes 0-3: button states
  data[0] = vRead(VBTN1);
  data[1] = vRead(VBTN2);
  data[2] = vRead(VBTN3);
  data[3] = vRead(VBTN4);
  // Bytes 4-5: POT1 (high byte, low byte)
  int pot1 = vAnalogRead(VPOT1);
  data[4] = highByte(pot1);
  data[5] = lowByte(pot1);
  // Bytes 6-7: POT2
  int pot2 = vAnalogRead(VPOT2);
  data[6] = highByte(pot2);
  data[7] = lowByte(pot2);

  CAN1.sendMsgBuf(MSG_ID, CAN_STDID, 8, data);

  Serial.print("TX: BTN=");
  Serial.print(data[0]); Serial.print(data[1]);
  Serial.print(data[2]); Serial.print(data[3]);
  Serial.print(" POT1="); Serial.print(pot1);
  Serial.print(" POT2="); Serial.println(pot2);
}
\end{lstlisting}

\begin{dicabox}
Abra duas janelas seriais: uma para o TX (onde você controla os botões e pots virtuais) e outra para o RX (onde você vê os dados recebidos via CAN). Isso simula um painel de controle completo sem acesso físico à placa.
\end{dicabox}

\subsection{Exemplo 3: Leitura mista (virtual + real)}

A biblioteca pode ser usada em conjunto com entradas físicas. Se o aluno tem acesso remoto à placa mas quer \emph{complementar} os sensores físicos com controles virtuais:

\begin{lstlisting}[language=C++,caption={Leitura mista: virtual + real}]
TASK(ReadInputs) {
  vioUpdate();

  // Botões: usa virtual se pressionado, senão lê o físico
  bool btn1 = vRead(VBTN1) || digitalRead(ECU1_DIN1);

  // Potenciômetro: soma virtual + real (com clamp)
  int pot = analogRead(ECU1_AIN1) + vAnalogRead(VPOT1);
  if (pot > 1023) pot = 1023;

  Serial.print("btn1="); Serial.print(btn1);
  Serial.print(" pot="); Serial.println(pot);
}
\end{lstlisting}

% === 16. Solução de Problemas ====================================
\section{Solução de Problemas}

\subsection{``make.py'' abre diálogo de seleção de programa}
O Windows não associou arquivos \texttt{.py} ao Python. Execute em um \textbf{Prompt de Comando} (CMD) como administrador:
\begin{codigobox}
assoc .py=Python.File\\
ftype Python.File="C:\textbackslash Users\textbackslash \%USERNAME\%\textbackslash AppData\\
\quad\textbackslash Local\textbackslash Programs\textbackslash Python\textbackslash Python311\\
\quad\textbackslash python.exe" "\%1" \%*
\end{codigobox}
Alternativa: sempre use \texttt{python make.py} em vez de \texttt{make.py}.

\subsection{``goil'' não é reconhecido como comando}
Abra um \textbf{novo} terminal após executar o Setup (as variáveis PATH são atualizadas no registro do Windows mas terminais já abertos não as recebem).

\subsection{VPN não conecta}
\begin{itemize}[nosep]
  \item Verifique suas credenciais do CIn
  \item Clique em ``Setup VPN Profile'' para recriar a conexão
  \item Verifique se outra VPN não está ativa
\end{itemize}

\subsection{Erro ao gravar firmware}
\begin{itemize}[nosep]
  \item Verifique se a porta COM está correta
  \item Tente resetar a placa antes de gravar
  \item Verifique se o arquivo \texttt{.hex} é válido
\end{itemize}

\subsection{Terminal serial: linhas quebradas ou caracteres estranhos}
\begin{itemize}[nosep]
  \item Verifique se o baudrate está correto (115200 para Trampoline)
  \item O log agora tem word wrap --- linhas longas (ex: frames CAN) são exibidas corretamente
\end{itemize}

\subsection{Setup: ``Python interpreter not found''}
O executável precisa de um Python instalado no sistema para executar o script de setup. Instale Python 3 e certifique-se de marcar ``Add to PATH'' durante a instalação.

\subsection{Setup: download falha com erro SSL}
Alguns servidores universitários têm certificados com problemas. O script automaticamente tenta novamente sem verificação SSL. Se ainda falhar, baixe o componente manualmente (o log indica a URL).
